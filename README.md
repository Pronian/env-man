# env-man

Layered environment and config file manager.

`env-man` generates the files outside `.env-man/` by stacking a **base layer**
with any number of named **overlay layers**. Mergeable file types (`.env`,
`.json`, `.yaml`, `.yml`) are deep-merged; everything else is overwritten by
the highest layer that contains it.

```
в””в”Ђв”Ђ config/

```

**Layers** are just folders under `.env-man/`. `base` is always applied at the
lowest priority and cannot be toggled off. Other layers apply in the order you
list them: leftmost = lowest priority, rightmost = highest.

**Merge rules** (per file, across layers, lowest в†’ highest):

| File type                                       | Behavior                                                          |
| ----------------------------------------------- | ----------------------------------------------------------------- |
| `.env`, `.env.*`                                | KEY=value pairs merged; higher layer wins for a key; normalized   |
| `.json`                                         | Deep object merge; arrays and scalars replaced                    |
| `.yaml`, `.yml`                                 | Deep mapping merge, base key order preserved; arrays replaced     |
| anything else (`.txt`, `.conf`, binaries, ...)  | Highest layer's bytes win                                         |

> `.env` values are treated **literally** вЂ” `$VAR` placeholders are never
> expanded. Output is normalized (sorted keys, quoting only where required).

## Quick start

```bash
env-man init                       # create the .env-man/ scaffold
# edit .env-man/base/.env, add .env-man/dev/.env, .env-man/local/.env, ...
env-man diff dev local             # dry-run: preview what apply would change
env-man apply dev local            # replace the stack and write files
env-man list                       # show the current stack and available layers
env-man                            # launch the interactive TUI
env-man drop                       # remove every file env-man generated
```

## Commands

| Command                | Description                                                            |
| ---------------------- | --------------------------------------------------------------------- |
| `env-man`              | Launch the interactive TUI (toggle/reorder layers, `enter` to apply).  |
| `env-man init`         | Create `.env-man/` with `state.yaml` and `base/.env` (heals gaps).    |
| `env-man apply [l...]` | Replace the layer stack with the given layers and write files.        |
| `env-man drop`         | Delete every file env-man materialized and clear the applied stack.   |
| `env-man list`         | Show the current stack and all available layer folders.               |
| `env-man diff [l...]`  | Dry-run preview of what `apply` would change (with per-key deltas).   |

## How apply works

`apply` writes files into the workspace **outside** `.env-man/` so that the
workspace reflects exactly the applied stack:

1. The stack is always `[base, ...layers]`.
2. Every file across all layers in the stack is collected; content is merged
   per the rules above.
3. Files are written atomically (temp-file + rename). Files that already exist
   are overwritten silently.
4. Files produced by a *previous* apply but no longer produced by the new
   stack (orphans) are removed, and empty directories are pruned.
5. `state.yaml` is updated with the new layer list and the manifest of
   materialized files.

`apply` replaces the stack вЂ” running `apply staging` when `[dev, local]` is
applied results in `[base, staging]`. Running `apply` with no arguments resets
to base only.

## `state.yaml`

```yaml
version: 1
layers:          # applied overlay layers, lowest priority first (base implicit)
  - dev
  - local
order:           # full display ordering of every known overlay layer
  - dev          # (including disabled ones); persisted by the TUI so disabling
  - staging      # a layer does not lose its position. `apply` overwrites this,
  - local        # keeping non-applied layers where they were.
  - prod
materialized:    # files env-man wrote outside .env-man/; used by `drop`
  - .env
  - config/db.yml
```

`order` is display metadata only вЂ” it does not affect merge priority (which
comes from `layers`). It exists so the TUI can remember the position of layers
you toggled off. When absent (e.g. a freshly initialized workspace), the TUI
falls back to listing applied layers first, then the rest alphabetically.

## Layer names

Layer folder names must match `^[a-zA-Z0-9_-]+$`. `base` is reserved.
`env-man apply` errors if a named layer's folder doesn't exist (it never
auto-creates layers).

## Discovery

`env-man` looks for `.env-man/` **only in the working directory** (or `--cwd`).
It does not search parent directories.

## TUI key bindings

| Key                       | Action                                            |
| ------------------------- | ------------------------------------------------- |
| `в†‘` / `в†“` or `k` / `j`    | Move the cursor                                   |
| `shift+в†‘` / `shift+в†“` or `K` / `J` | Reorder the selected layer (changes priority) |
| `space`                   | Toggle the layer on / off                         |
| `enter`                   | Apply the current selection and exit              |
| `q` / `esc` / `ctrl+c`    | Quit without applying                             |

The list runs low в†’ high priority top to bottom (base is locked at the top).
Moving a layer down raises its priority (it's applied later, overriding more).
The full ordering вЂ” including layers you toggle off вЂ” is persisted to `order` in
`state.yaml` on apply, so disabling a layer never loses its place. Running
`env-man apply` overwrites `order`: applied layers are re-sequenced into their
CLI priority order within their existing slots, while layers `apply` doesn't
touch keep their previous positions.
